Архітектурні рішення
Проєкт використовує кілька ключових патернів:

Repository Pattern: Ізолює логіку роботи з файлами від бізнес-логіки. Ви можете змінити файлове сховище на базу даних, не чіпаючи сервіси.

Strategy Pattern: Аналітичні розрахунки винесені в окремі класи стратегій. Це дозволяє легко додавати нові типи аналітики (наприклад, прогнозування чи тренди), не змінюючи код HealthService.

Dependency Injection: Впровадження залежностей дозволяє легко підміняти реальні репозиторії на "моки" під час тестування.

 Робота системи
Коли користувач вводить дані в інтерфейс Flet:

UI передає дані в Service Layer.

Service Layer валідує дату (дозволено лише поточний день) та створює об'єкт HealthEntry.

Repository серіалізує об'єкт у JSON і зберігає його на диск.

При запиті аналітики, Strategy обробляє весь масив записів і повертає математичний результат.

Тестування
Ми приділяємо велику увагу якості. Кожен метод покритий тестами:

Unit-тести: Для логіки менеджерів та сервісів (використовуючи pytest та unittest.mock).

File-тести: Для перевірки цілісності даних при роботі з JSON.

Bash
# Запуск тестів з генерацією звіту про покриття
